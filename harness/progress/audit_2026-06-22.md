### ⚠️ Reglas Núcleo

**Status:** ⚠️ **2 HALLAZGOS MENORES**

#### Hallazgo 1: Acceso directo a `process.env` en Playwright config

**Archivo:** `apps/web/playwright.config.ts:12-14`  
**Violación:** Convención "Variables de entorno solo vía esquema validado" (nunca `process.env` directo)  
**Severidad:** MENOR (config de herramienta, no código de app)  
**Código:**
```typescript
const isCi = Boolean(process.env.CI);
const webPort = Number(process.env.WEB_PORT ?? DEFAULT_WEB_PORT);
const apiPort = Number(process.env.API_PORT ?? DEFAULT_API_PORT);
```
**Remediación sugerida:** Crear `PlaywrightConfig` schema en Zod (packages/schemas) y validar aquí. O: aceptar pragmáticamente que archivos de config de herramientas (Playwright, Vite, etc.) pueden acceder directo a env con defaults (riesgo bajo, no toca datos de dominio).  
**Criterio para decidir:** Depende del proyecto. Si es una "regla núcleo dura", requiere remediar; si es flexible para tooling, puede dejarse. **Propuesta:** Dejar como está (pragmático), documentar en conventions.md que "archivos de config de herramientas (.config.ts) pueden acceder a env con fallbacks seguros".

#### Hallazgo 2: TODO pendiente sin tracking en backlog

**Archivo:** `apps/web/e2e/fixtures.ts` (línea ~)  
**Violación:** Convención "Sin TODOs/FIXMEs pendientes en código (o documentados en backlog)"  
**Severidad:** MENOR (nota de implementador, no bloquer)  
**Código:**
```typescript

```
**Remediación sugerida:** Convertir a feature de backlog (`POLISH-001: E2E test fixtures auth automation`) o cerrar el TODO si está en scope de MACHINES-001 (el contexto sugiere que se resuelve cuando se implemente el primer endpoint protegido).
